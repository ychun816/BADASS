# P1 

## prerequisit install | install GNS3 (in `root`)

- official site: https://docs.gns3.com/docs/getting-started/installation/linux/

- The GNS3 Server is a Python application. There are no official .deb packages or AppImages for the server on Debian, and the Ubuntu PPA does not work on Debian.
- Installing with pip in a virtual environment is the safest way to get the latest GNS3 Server on Debian, without breaking system Python or package dependencies.
- The GNS3 GUI is a desktop app, distributed as an AppImage, which is easy to run on any Linux system without installation.
>  The GUI makes it much easier to interact with GNS3 projects, drag-and-drop devices, and connect them.
> If only use the GNS3 Server, you would have to interact with it via API or command line, which is not practical for most users. The GUI is the standard way to use GNS3 for network simulation.

-  cannot just run the GNS3 AppImage alone BECUZ:
    - AppImage is only the GUI (client). It needs to connect to a running GNS3 Server, which must be installed and started separately.
    - The server handles all the backend emulation and networking. The GUI is just the frontend.
>  need both:
> - The server (via pip/venv)
> - The GUI (AppImage)
> -> the most reliable and supported way for Debian users!

```bash
# sudo usermod -aG sudo <USER NAME>
sudo usermod -aG sudo yilin

# exit and re-ssh into vm
exit

# SSH back into the VM as <USER NAME>
ssh yilin@127.0.0.1 -p 2223
```

## verify gns3 server & GUI running (after launching gns3 in vm)
> confirm GUI and server are running, listening, and healthy.
```bash
# launch gns3
gns3

# Check the server process
ps aux | grep gns3server

# Check the listening port
sudo ss -tulnp | grep 3080

# Optional – Test API via Python (more reliable than curl)
python3 -m pipx run gns3server --version

# Optional – view latest server log entries
tail -f ~/.config/GNS3/3.0/gns3_server.log

```

# core concepts

| Concept                           | Explanation                                                                           | Why It Matters in BADASS                                        |
| --------------------------------- | ------------------------------------------------------------------------------------- | --------------------------------------------------------------- |
| **BGP (Border Gateway Protocol)** | EGP (Exterior Gateway Protocol) that exchanges routes between Autonomous Systems (AS) | Needed to simulate AS-level routing between your Docker routers |
| **Autonomous System (AS)**        | A network under a single administrative domain with a unique AS number                | Each router in your lab will belong to an AS (e.g., 65001)      |
| **eBGP vs iBGP**                  | eBGP – between different AS, iBGP – inside same AS                                    | Determines neighbor configuration in your topology              |
| **Neighbor / Peer**               | BGP neighbor router with which you exchange routes                                    | Must configure properly in `bgpd` for route propagation         |
| **AS Path**                       | List of AS numbers a route has passed                                                 | Helps prevent routing loops; shows the route path               |
| **Next-Hop**                      | IP address for packet forwarding                                                      | Must be reachable for routes to work                            |
| **BGP FSM States**                | Idle → Connect → Active → OpenSent → OpenConfirm → Established                        | Shows neighbor negotiation status; “Established” is required    |
